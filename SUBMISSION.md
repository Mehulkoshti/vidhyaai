## Additional Question: "If you have used Sarvam to build your project, explain how you used it in detail"

```
Sarvam AI is the primary AI engine of VidhyaAI — we use three Sarvam products
(with a Google Gemini fallback for reliability, described at the end):

1) CHAT COMPLETIONS (sarvam-30b) — Every study mode is generated by Sarvam's LLM
   via the OpenAI-compatible /v1/chat/completions endpoint, with reasoning_effort=low
   and a terse "return only JSON" instruction so the model emits clean, parseable
   JSON that we validate ourselves. It powers summaries, quizzes, flashcards,
   ELI5/detailed/analogy explanations, mind maps, mock tests (with per-question
   "concept" tags used to flag weak areas), day-by-day study planners, full course
   outlines, and a context-aware doubt-solving tutor chat. We use retry/backoff for
   transient errors. The user picks the output language (English, Hindi, Hinglish,
   Marathi, Tamil, Bengali and more) and Sarvam's India-first training makes the
   regional-language output genuinely natural — this is the core reason we chose
   Sarvam as our primary LLM.

2) BULBUL v3 TEXT-TO-SPEECH — The "Listen aloud" button sends the generated text to
   Sarvam's Bulbul TTS (target_language_code mapped from the chosen language) and
   plays back natural, native-sounding Indian voices — a huge upgrade over robotic
   browser TTS for revision on the go. Audio is cached per text to avoid re-billing.

3) SARVAM VISION (Document Intelligence) — When a student attaches a photo or PDF of
   their notes, we run Sarvam Vision OCR (create-job → upload → start → poll →
   download) to extract the text, then feed it straight into the study-material
   prompt. Images are wrapped into a single-page PDF before upload.

RELIABILITY FALLBACK (Google Gemini) — Sarvam is primary and gets first shot at
every request. On Sarvam's free "starter" tier the LLM has a 4096-token output cap
that heavier requests (longer mock tests, day-by-day planners, full course outlines)
can exceed. To keep the app rock-solid, if a Sarvam call fails or times out we
transparently fall back to Google Gemini (gemini-2.5-flash) for that request,
asking for the exact same JSON shape — so the experience is seamless, the user
always gets a result, and no mode ever breaks. Sarvam remains the primary engine
and does the visible work in the demo; Gemini is a silent safety net.

Architecture: Next.js 16 App Router; all Sarvam (and fallback Gemini) calls are
proxied through server API routes so neither API key ever reaches the browser. A
built-in mock mode let us build the entire UI without spending credits, and
TTS/Vision are gated behind explicit user actions to keep credit usage minimal.
```

## Additional Question: "If you have used Expo to build your mobile App, explain how you used it in detail"

```
We built a native VidhyaAI mobile app with Expo (SDK 57) + React Native + TypeScript
(in the /mobile folder of the repo). It's a thin native client that reuses the exact
same deployed backend as the web app — it calls our /api/generate endpoint, so all
Sarvam AI calls stay server-side and no API key ever ships inside the app.

The app lets students pick a study mode (Summary, Explain, Quiz, Flashcards, Mind Map,
Planner) and any of 12 languages, type a topic, and get AI study material rendered
natively: interactive quizzes (tap an option to reveal the answer), tappable flip
flashcards, mind-map branches, and day-by-day planners — in a dark, branded UI.

Expo tooling: developed and previewed with `npx expo start` on Expo Go, bundled with
Expo's Metro exporter, and packaged into an installable Android APK via EAS Build
(`eas build -p android`). Expo let one developer ship a real cross-platform mobile
app on top of the existing web backend in a fraction of the usual native effort.
```
